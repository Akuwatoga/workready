# Design: WorkReady Analyst Wedge

Generated by office-hours on 2026-06-30
Status: DRAFT
Mode: Intrapreneurship + Open Source

## Problem Statement

WorkReady is currently a general workplace skill pack for students, interns, new graduates, and new employees. That positioning is directionally right but too broad for a pre-product project.

The first wedge should be narrower: help a 23-year-old data analyst new graduate in a large company survive and succeed during a 3-6 month probation period.

The core problem is not that this user lacks career advice. They can already find advice on Xiaohongshu, Zhihu, public articles, mentors, and ChatGPT. The real problem is that fragmented advice does not become an operating workflow when the user receives a real business analysis task.

WorkReady should convert workplace advice into concrete artifacts:

- requirement clarification
- metric definition
- data source map
- analysis plan
- manager/requester confirmation message
- progress update
- meeting action items
- weekly evidence log
- probation/promotion proof package

## Demand Evidence

The current evidence is pain-point evidence, not adoption evidence.

Observed and researched signals:

- Xiaohongshu-style content repeatedly advises new employees to confirm probation goals, clarify task requirements, report progress regularly, and summarize conclusions before details.
- Common posts around "工作能力差", "实习生工作能力差", "试用期", "职场新人", and "工作 SOP" cluster around the same operating gaps: task confirmation, active reporting, file/task organization, communication, and review habits.
- Public career guidance for new employees and interns repeatedly emphasizes communication, execution, clear writing, learning ability, and regular feedback.
- HR/management content around probation emphasizes clear role expectations, job descriptions, regular tracking, mentors, and evaluation evidence.

This supports the conclusion that the pain is real and frequent. It does not yet prove that users will adopt WorkReady. The next validation must test whether target users use the artifact in real work.

## Status Quo

The target user currently solves this badly through:

- searching Xiaohongshu/Zhihu/public posts and saving screenshots
- asking friends, classmates, mentors, or senior colleagues
- using ChatGPT for one-off messages or weekly reports
- copying templates from notes apps
- learning by trial and error after leader criticism or task rework

The status quo is not "no solution." It is fragmented advice plus painful feedback loops.

Cost to the new analyst:

- repeated rework from unclear scope or metric definitions
- anxiety about whether the leader thinks they are capable
- invisible work because progress and impact are not reported clearly
- weak probation evidence because daily work is not logged into a coherent record

Cost to the leader/mentor:

- repeated explanation of basic work habits
- reviewing messy outputs that lack assumptions, conclusion, or business recommendation
- difficulty distinguishing attitude problems from method problems
- late discovery of blockers because the new employee does not report early

Cost to the organization:

- onboarding depends on individual mentor quality
- probation evaluation lacks process evidence
- good operating habits are not standardized or reused
- training content is disconnected from live work

## Target User & Narrowest Wedge

Primary target user:

> A 23-year-old data analysis new graduate in a large company, working through a 3-6 month probation period in a business data, operations analytics, strategy analytics, or BI team.

What gets them recognized:

- understanding the business problem, not just pulling data
- confirming scope, metric definitions, data sources, timeline, and deliverable before starting
- producing clear conclusions and recommendations, not only tables and charts
- reporting progress and blockers before the leader asks
- keeping a visible record of tasks, meetings, decisions, data sources, and weekly learning

What puts them at risk:

- unclear metric definitions
- silent work with no progress update
- charts without business conclusion
- missing requester/decision-maker/stakeholder map
- weak meeting follow-up
- no evidence package at probation review time

Narrowest wedge:

> Analyst Intake: when the new analyst receives a vague business/data analysis request, WorkReady generates a requirement clarification checklist, metric/data-source risk map, analysis plan, and confirmation message in 3 minutes.

This is the first daily-use workflow. It naturally feeds future modules:

```text
/analyst-intake -> /datamap -> /analysis-plan -> /update -> /retro -> /promotion-pack
```

## Premises

1. WorkReady should not start with "all workplace newcomers." It should first win with "new data analysts in probation."
2. The first wedge should not be the full workplace operating system. It should be the high-frequency moment where a new analyst receives a vague data/business request.
3. Xiaohongshu/Zhihu content is valuable as demand research and language mining, but it is not the product form.
4. The product form is a workflow that converts raw task context into executable artifacts.
5. `/first90`, `/update`, `/retro`, and `/datamap` should orbit the analyst intake wedge before expanding to other job families.

## Approaches Considered

### Approach A: Analyst Wedge First

Focus WorkReady on the analyst intake workflow first. Rewrite the most relevant skills around the analyst probation journey.

Effort: M
Risk: Low

Pros:

- concrete user and job-to-be-done
- high-frequency usage moment
- clear before/after quality improvement
- strong fit with existing `/plan`, `/datamap`, `/update`, `/meeting`, `/retro`

Cons:

- narrower initial audience
- requires analyst-specific examples, terminology, and templates
- less impressive than a broad "workplace OS" story at first glance

### Approach B: WorkReady Lifecycle System

Build the complete workplace lifecycle from day one: onboarding, first 90 days, planning, RACI, meeting, status, retro, and promotion pack.

Effort: L
Risk: Medium

Pros:

- closer to gstack's full lifecycle feel
- stronger open-source positioning
- covers multiple user types

Cons:

- too broad for pre-product
- weak validation loop
- users may not know where to start

### Approach C: Training Kit For Teams

Position WorkReady first for leaders, mentors, HR, or training teams who need to onboard and evaluate new employees.

Effort: M/L
Risk: Medium

Pros:

- stronger intrapreneurship fit
- closer to organizational adoption
- can standardize manager/mentor behavior

Cons:

- farther from the user's daily pain
- requires more organizational process design
- slower adoption in open-source self-serve use

### Approach D: Analyst Wedge, Then Lifecycle And Team Kit

Start with Approach A, but design the architecture so it can later grow into Approach B and C.

Effort: M now, L over time
Risk: Low/Medium

Pros:

- preserves focus now
- keeps the long-term WorkReady vision
- creates a path from personal usage to team adoption
- aligns with gstack's wedge-first then lifecycle-stack philosophy

Cons:

- requires discipline not to build every skill equally at first
- needs a roadmap that prevents "generic workplace advice" from creeping back in

## Recommended Approach

Choose Approach D, with Approach A as the immediate implementation strategy.

The v1 should be:

> WorkReady Analyst: an AI skill stack that helps a new data analyst turn vague business requests into clear requirements, analysis plans, progress updates, and probation evidence.

The broader WorkReady system should remain visible but secondary.

## Optimized Product Shape

### Core Lifecycle

```text
Intake -> Clarify -> Map Data -> Analyze -> Update -> Review -> Evidence
```

### Primary Skills

| Skill | Role | Output |
|---|---|---|
| `/analyst-intake` | Convert vague analysis request into scoped work | requirement checklist, assumptions, metric questions, confirmation message |
| `/datamap` | Map source systems and data risk | source map, metric owner, quality issues, access gaps |
| `/analysis-plan` | Turn request into analytical plan | hypothesis, method, data pull plan, deliverable outline |
| `/update` | Report progress and risk | manager/requester update |
| `/meeting` | Convert syncs into decisions/actions | decisions, owners, deadlines |
| `/retro` | Turn weekly work into learning | weekly evidence, lessons, next week priorities |
| `/promotion-pack` | Assemble probation evidence | impact log, skill growth, projects, feedback, next plan |

### Supporting Skills

| Skill | Role |
|---|---|
| `/first90` | probation strategy |
| `/raci` | cross-functional ownership |
| `/stakeholder` | requester/decision-maker/user map |
| `/message` | workplace communication variants |

## What To Borrow From gstack

Do not copy gstack's surface. Borrow its mechanisms:

1. Lifecycle, not command list
   - gstack: think -> plan -> review -> QA -> ship -> retro
   - WorkReady: intake -> clarify -> map -> analyze -> update -> review -> evidence

2. Design docs feed downstream work
   - WorkReady should save probation plans, analysis plans, and promotion evidence under `memory/` or a user workspace.

3. Review gates
   - Before an analysis plan is considered ready, run a checklist: metric definition, data source, stakeholder, deliverable, risk, decision owner.

4. Sections and manifests
   - Large skills should be split into sections so prompts stay maintainable.

5. First-run routing
   - New users should not see 12 commands. They should answer: "Are you starting a new role, receiving a data task, preparing an update, or building a promotion package?"

6. Team mode
   - Later: allow a team to install WorkReady templates and routing into their repo/workspace for shared onboarding.

7. Memory/learning loop
   - The system should remember recurring manager preferences, metric definitions, common data sources, and user's weekly lessons.

8. Quality gates, not advice
   - Every skill should end with an artifact and a readiness check.

## Proposed Roadmap

### Phase 1: Analyst Wedge

Goal: make one workflow excellent.

Build:

- `/analyst-intake`
- `/analysis-plan`
- analyst-specific `/datamap`
- analyst-specific templates
- examples based on realistic business analysis requests
- README repositioning around the wedge

Success criteria:

- A new analyst can process a vague request in under 5 minutes.
- The output includes scope, metrics, data source, deliverable, risks, and confirmation message.
- The user can paste the confirmation message to a leader/requester with minimal edits.

### Phase 2: Probation Evidence Loop

Goal: convert daily work into probation proof.

Build:

- `/promotion-pack`
- weekly evidence log template
- first 90 days analyst plan
- manager update loop

Success criteria:

- User can produce a weekly manager update from task records.
- User can produce a probation review packet from accumulated evidence.

### Phase 3: Team Training Kit

Goal: make this useful for leaders, mentors, HR, and internal training.

Build:

- mentor guide
- leader review checklist
- probation tracking template
- team install/routing docs

Success criteria:

- A leader can use WorkReady to guide a new analyst without writing custom instructions.
- HR/mentor can see consistent evidence across new hires.

### Phase 4: Open Source Skill Platform

Goal: make WorkReady extensible beyond analysts.

Build:

- contribution guide
- skill authoring guide
- role packs: analyst, operations, product, marketing, customer success
- install script / host adapter documentation

Success criteria:

- Contributors can add a new role pack without changing the core router.
- Users can install only the role pack they need.

## Open Questions

1. Should the first public README lead with "WorkReady" broad vision or "WorkReady Analyst" wedge?
2. Should `/analyst-intake` be a new skill or should `/plan` become analyst-aware?
3. Should memory live inside the repo `memory/` for this project, or should user-generated work artifacts live outside the repo?
4. Which platform should be the first install target: Codex, Claude Code, Cursor, or generic Markdown?
5. What is the first validation test: ask 5 analysts to use `/analyst-intake`, or publish a Xiaohongshu-style demo and measure saves/comments?

## Success Criteria

Product validation criteria:

- 5 target users try `/analyst-intake` on a real or recent data-analysis task.
- At least 3 say the output would have reduced rework or improved confidence.
- At least 2 use the confirmation message or plan format in real work.
- At least 1 leader/mentor says the output would make a new analyst easier to manage.

Artifact quality criteria:

- Every analyst intake output includes:
  - business background
  - requester/decision owner
  - metric definitions
  - data source and owner
  - assumptions
  - risks
  - deliverable format
  - timeline
  - confirmation message

## Dependencies

- Need a small corpus of real-world analyst tasks and Xiaohongshu/Zhihu language patterns.
- Need examples for business data, operations data, product data, and BI/reporting tasks.
- Need a decision on whether WorkReady optimizes first for English, Chinese, or bilingual usage.
- Need a contribution structure before inviting open-source role packs.
